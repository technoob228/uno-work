Михаил хочет фичу **агент-супервизор (manager/supervisor agent)** в uno-work-app: отдельный персистентный чат, который не исполняет задачи сам, а оркеструет остальные чаты-харнессы (Claude/Codex/Cursor/OpenCode), и выводится во внешний чат (Telegram) для верхнеуровневого управления проектом с телефона.

**Сценарий:** запустил несколько окон (Claude, Codex) с задачами, ушёл, с телефона через Telegram спросил «что там с задачами, которые я пораздавал» — менеджер отвечает статусами; может сам создать/продолжить тред и раздать задачу.

**Ключевое архитектурное решение:** менеджер НЕ глотает сырые контексты дочерних чатов (иначе тот же лимит контекста + компакт). Он читает **проекции** (`ProjectionThreadSessions`, `ProjectionSnapshotQuery`) — компактные сводки статусов, а детали тянет по запросу как tool. «Не сбрасывать контекст» = работа со сводками, а не бесконечный контекст.

**Сборка из 4 кусков:** (1) менеджер-тред / новый `MetaDriver`; (2) tool-слой оркестрации тредов (list/get_status/read/create/send/interrupt) — тонкие обёртки над существующими командами (`ProviderCommandReactor`, `ProviderService`); (3) Telegram ingress/egress поверх `RuntimeReceiptBus` + проактивные пуши; (4) политика подтверждений.

**Риски:** автономные write-действия из Telegram (нужен read-only по умолчанию + подтверждение на запись); жёсткая аутентификация Telegram (только свой chat_id); держать менеджера тонким диспетчером, тяжёлое — в дочерних тредах.

## Ремарки из видео Theo "I guess we're writing loops now" (резонируют в разные стороны)

**Что ВАЛИДИРУЕТ наш план (мы пришли независимо):** Pete's loop «wake up every 5 min, direct work to threads» = буквально наш менеджер + heartbeat; Codex stock уже умеет тред→спавнит тред (`create_thread` не теория); «Hermes brings context to me instead of going out» = ровно наш проактивный менеджер; «SSHing to another machine on my network running this work tree, monitor PR 6+ hours» = наш remote-исполнитель на демоне + изоляция.

**Что ПОДСВЕЧИВАЕТ (явно дописать в дизайн):** (1) механизм пробуждения = **heartbeat/polling** (wake every 5-10 min, читает статусы тредов) — раньше говорили «персистентный», теперь конкретно heartbeat; (2) **work trees** как дешёвый механизм изоляции исполнителей (дополняет контейнеры, особенно локально); (3) **авто-ревью-петля** («looking at code too early» — агенты ревьюят ДО человека) как ядро ценности: тебе прилетает «готово и проверено», не черновик; (4) два режима исполнения — `/goal` (линейный вечный цикл до завершения одной задачи) vs **dynamic workflow** (многоэтапные стэкнутые PR, под-ревью, loops-creating-loops).

**НАПРЯЖЕНИЯ (тянет в обе стороны — тут принимать осознанные решения):**

- **Автономность ↔ контроль.** Видео соблазняет полной автономией («проснулся — 4 PR смержены»). Но Theo сам НЕ на fully-autonomous, и для UNO (прод + деньги клиентов) автономия = риск. Решение: автономия по умолчанию на dev/изолированном, надзор обязателен на проде.
- **Динамическая форма ↔ детерминированная надёжность.** Theo за динамические loops, против хардкод-персон («predefining personas misses the point — agent builds context dynamically»). Но полная динамика = непредсказуемость, тяжело гарантировать на проде. Решение: динамика для исследования/разработки, ДЕТЕРМИНИРОВАННЫЕ воркфлоу для критичных повторяемых операций (деплой, миграция, изменение баланса).
- **«Не смотри код рано» ↔ ответственность за прод.** Делегировать ревью агентам экономит время, но на проде с балансами юзеров человек ОБЯЗАН быть на gate. Решение: авто-ревью + человек-gate именно на prod-merge/опасных действиях.
- **Hermes: adopt ↔ build.** Theo активно юзает Hermes для context-bringing → соблазн взять готовое. Но у вас ~60% нативно + Hermes жрёт >1GB RAM (Theo переписывает его на Rust). Решение остаётся: паттерн берём, код — нет, только референс.

**КРАСНЫЙ МАРКЕР для UNO (главное отличие от Theo):** Theo сам говорит «не на проде с миллионами юзеров», гоняет на pet-проектах через личные $200-подписки. Наш менеджер трогает ПРОД fishcode + балансы + VPS клиентов → подтверждения/read-only-по-умолчанию/аудит в `logs` — НЕ опция, а несущая конструкция.

**ЭКОНОМИКА loops (бизнес-критично, у Theo скрыто subscription'ами):** loops жгут токены безумно — 10-мин ревью развернулся в 8ч workflow на 3M токенов; Theo ~$10k/мес инференса, но через личные подписки, не API. «If you're paying API prices, you probably shouldn't be doing loops yet.» Для UNO: если инференс идёт через вашу инфру/прокси — кто платит за 8-часовой автономный цикл клиента? Без бюджетов/лимитов на автономные loops маржа умрёт. → роутинг дешёвых моделей на рутину = УСЛОВИЕ ВЫЖИВАНИЯ экономики, не оптимизация. Это самый недопроработанный угол.

Фундамент готов ~на 60% (event-sourced оркестрация уже есть). Hermes (Nous Research) для этого НЕ нужен — паттерн реализуется нативно в Effect-архитектуре. См. [[project_uno_search_integration]] (другой Electron→Gateway мост) и роутинг моделей по типу задачи (оркестрация/тяжёлое/картинка/видео/рутина).

**Доступы per-тред:** `ProviderSessionStartInput` уже несёт `cwd` + `modelSelection`; есть `runtimeMode`→permissionMode (`full-access`→bypassPermissions) и permission-callback через Claude SDK. Напрашивается слой «профилей доступа» {cwd allowlist, permissionMode, allowedTools, mcp, network}. Доступы НЕ наследуются от менеджера к дочернему треду (изоляция) → нужен least-privilege: менеджер раздаёт только подмножество своих прав. «Инет» — не один флаг: web-tools/MCP закрываются permission-callback, но при разрешённом bash агент выйдет через curl → настоящая сетевая изоляция = sandbox/контейнер ОС, не флаг. Менеджер тонкий (диспетчер), исполнение — через дочерние треды.

**Топология (решение Михаила, июнь 2026):** менеджер живёт ТАМ ЖЕ, где исполнители — единый **рантайм-демон** (менеджер LLM + исполнители в контейнерах + durable журнал). Electron UI и Telegram-бот — оба тонкие клиенты, цепляются к демону по одному resumable-протоколу; «связка в ТГ с локальным/удалённым» = выбор эндпоинта демона. Это убирает хрупкость с критического пути: менеджер↔исполнители локально внутри машины (без интернета), хрупким остаётся только ты↔менеджер (Telegram/UI), а он терпит обрывы (агенты продолжают, события копятся, клиент догоняет). Демон где угодно — один движок в двух местах; начать с демона на VPS, локальный добавить тем же кодом для оффлайн/локального кода. Открытая развилка: локальный комп = только вьюпорт или тоже место исполнения (тогда нужен и локальный демон). `apps/server` уже почти этот демон.

**Большое видение (полный контур):** менеджер раздаёт задачи в т.ч. в **удалённые окружения на VPS** (исполнитель крутится на сервере рядом с кодом, Electron — тонкий клиент). 5 этапов в безопасном порядке: (1) security-фундамент — SSH по ключам вместо plain-text root-паролей в servers\*.csv, не-root, контейнеризация исполнителя; (2) resumable транспорт local↔server (WebSocket/gRPC + reconnect, догон событий после обрыва через event-sourcing — лечит боль vm-pico из [[project_remote_environments]]); (3) remote execution; (4) sync кода git-first + опционально mutagen; (5) Telegram наружу — последним (самый опасный путь к проду: авторизация по chat_id + подтверждение write + аудит в logs). Security здесь определяет архитектуру, а не довесок.
